PART 1 - PROJECT OVERVIEW
================================================================================

WHAT THIS IS
------------
A private, friend-group-only web app replicating the IPL auction experience
for EA FC 26 fantasy team building.

- NOT a public marketplace
- NOT a commercial product
- NOT a real-money platform

CORE EXPERIENCE ANALOGY
------------------------
Player Discovery    = EA FC Ratings website - search, filter, cards, full stats
Auction System      = IPL auction - live bidding, timer, budgets, sold/unsold
Team Management     = Fantasy Premier League - squad, budget tracker, rankings
Admin Controls      = Broadcast studio - one admin controls all states

USER ROLES
----------
Role          Count       Capability                      Login
-----------   ---------   -----------------------------   --------
Admin         1           Full control, create/run all    Required
Team Owner    8-12        Bid and manage squad            Required
Guest         Anyone      Watch only                      Not required

PROBLEMS THIS SOLVES
---------------------
- No Excel sheets, everything inside one platform
- No manual player entry, import from community FC 26 JSON dataset
- No tab switching, search, auction, budgets all in one screen
- Real-time, every bid visible to all owners simultaneously
- Mobile-first bidding, owners bid on phones, admin on desktop

PART 7 - AUTHENTICATION MODULE
================================================================================

JWT FLOW
---------
1. User submits email and password to POST /api/auth/login
2. Server validates, returns JWT token (expires 7 days)
3. Frontend stores token in localStorage under key fc26_token
4. All API requests include: Authorization: Bearer TOKEN
5. Socket.IO connection includes token in auth handshake: { token }
6. Server middleware decodes token, attaches req.user to every request

ROLE-BASED ROUTE PROTECTION
-----------------------------
Public routes       /login, /register, /guest, no token required
Owner routes        Token required + role must be owner or admin
Admin routes        Token required + role must be admin only
Socket connection   Token validated on connect, role determines allowed events

NEXT.JS MIDDLEWARE
-------------------
Create frontend/src/middleware.ts
Reads JWT from cookies, checks role, redirects unauthenticated to /login.

LOGIN PAGE RULES
-----------------
- Single login form for all roles, role determined by JWT payload
- On success: redirect to /dashboard (role-specific routing via middleware)
- Show error toast on invalid credentials
- Do not reveal whether email or password is wrong
- Auto-redirect if already logged in

PART 11 - LIVE AUCTION AND SOCKET.IO ARCHITECTURE
================================================================================

IMPORTANT: Read this entire section before writing any Socket.IO code.

AUCTION STATE MACHINE
----------------------

pending --> active --> (per player: bidding --> sold or unsold) --> completed
                 |
              paused

State           Meaning                             Transition
-----------     --------------------------------    ---------------------------
pending         Created, not started                Admin clicks Start = active
active          Player on block, timer running      Timer expires = sell/unsold
paused          Admin paused                        Admin resumes = active
completed       All pool players done               Terminal, no transitions out

SOCKET.IO ROOM ARCHITECTURE
-----------------------------
Room name           auction:AUCTION_ID
Who joins           Everyone, admin, owners, guests
Admin extra         Also joins admin:AUCTION_ID
Private events      Budget updates sent to team:TEAM_ID
Broadcast events    Bids sent to entire auction room

CLIENT TO SERVER EVENTS (What Clients EMIT)
--------------------------------------------
Event               Payload                                 Permission
----------------    ------------------------------------    ----------------
join_auction        { auctionId, token }                    Everyone
place_bid           { auctionId, amount, teamId, token }    Owners only
admin:start         { auctionId }                           Admin only
admin:pause         { auctionId }                           Admin only
admin:resume        { auctionId }                           Admin only
admin:skip          { auctionId }                           Admin only
admin:unsold        { auctionId }                           Admin only
admin:undo          { auctionId }                           Admin only
admin:next_player   { auctionId }                           Admin only

SERVER TO CLIENT EVENTS (What Server EMITS)
--------------------------------------------
Event               Payload
----------------    -------------------------------------------------------
auction:state       Full auction state on join or change
bid:new             { amount, teamName, teamId, timestamp }
bid:winning         { amount, teamName }
timer:tick          { seconds } emitted every second
timer:warning       { seconds: 5 } at 5 seconds remaining
player:sold         { player, team, price }
player:unsold       { player }
player:next         { player }
auction:paused      {}
auction:resumed     {}
auction:completed   { summary }
budget:updated      { teamId, remaining } private to team room only
error               { message, code } only to bidder

BID VALIDATION - 10 STEPS (Server-Side, ALL REQUIRED)
-------------------------------------------------------
1.  Check auction.status equals active
2.  Check playerId matches auction.currentPlayerId
3.  Check amount > auction.currentBid + auction.minBidIncrement
4.  Check team.remainingBudget >= amount
5.  Check team is part of this auction
6.  Check bidder is not current highest bidder (no self-overbid)
7.  findOneAndUpdate Auction with conditional { currentBid: { $lt: amount } }
8.  If update returns null, bid lost race condition, emit error to bidder only
9.  Reset timer to timerDuration
10. Emit bid:new to all room participants

IMPORTANT: ALL bid validation happens on the server. Never trust client amounts.

RACE CONDITION PROTECTION
--------------------------
Use MongoDB conditional findOneAndUpdate:

  const updated = await Auction.findOneAndUpdate(
    { _id: auctionId, currentBid: { $lt: newBidAmount } },
    { $set: { currentBid: newBidAmount, highestBidderId: teamId } },
    { new: true }
  );

  if (!updated) {
    socket.emit('error', { message: 'Bid was outpaced', code: 'BID_RACE_LOST' });
    return;
  }

Do NOT use separate read-then-write, that creates a race condition.

TIMER ARCHITECTURE
-------------------
- Timer runs on the SERVER, not the client
- Server uses setInterval stored in Map<auctionId, NodeJS.Timeout>
- Every second: emit timer:tick to room
- On bid received: clearInterval, restart timer, emit new tick
- Timer hits 0 with currentBid > 0: trigger sell logic
- Timer hits 0 with currentBid = 0: auto mark unsold
- Always clearInterval on: pause, sell, skip, unsold, complete

PLAYER SOLD LOGIC (Server, Step by Step)
-----------------------------------------
1. clearInterval for this auction
2. Player.findOneAndUpdate: status=sold, soldTo, soldPrice, soldAt=now
3. Team.findOneAndUpdate:
     $inc remainingBudget by -soldPrice
     $inc squadSize by +1
     $inc teamValue by +soldPrice
     $push players: playerId
4. Create BidHistory entry with isWinningBid: true
5. Emit player:sold to entire auction room
6. Emit budget:updated to team:TEAM_ID private room
7. Move to next player in pool or emit auction:completed

UNDO LAST SALE (Server, Step by Step)
---------------------------------------
1. Find most recent sold player in this auction
2. Player: status=pool, soldTo=null, soldPrice=null, soldAt=null
3. Team: $inc remainingBudget by +soldPrice
         $inc squadSize by -1
         $inc teamValue by -soldPrice
         $pull players: playerId
4. Delete all BidHistory entries for that player in this auction
5. Put player back at front of auction queue
6. Emit auction:state to all, everyone sees the reversal

NOTE: Only the most recent sale can be undone. Cannot skip backwards.

SOCKET.IO SERVER SETUP
------------------------
backend/src/socket.ts

  const io = new Server(httpServer, {
    cors: { origin: process.env.FRONTEND_URL, credentials: true }
  });

  // Middleware, validate JWT on every connection
  io.use((socket, next) => {
    const token = socket.handshake.auth.token;
    if (!token) return next(new Error('Authentication error'));
    try {
      const decoded = jwt.verify(token, process.env.JWT_SECRET);
      socket.data.userId = decoded.id;
      socket.data.role = decoded.role;
      socket.data.teamId = decoded.teamId;
      next();
    } catch {
      next(new Error('Invalid token'));
    }
  });

  io.on('connection', (socket) => {
    socket.on('join_auction', ({ auctionId }) => {
      socket.join('auction:' + auctionId);
      if (socket.data.role === 'admin') socket.join('admin:' + auctionId);
      if (socket.data.teamId) socket.join('team:' + socket.data.teamId);
    });
  });

PART 13 - SOLD PLAYER AND BUDGET MANAGEMENT
================================================================================

All budget deductions and player assignments happen automatically on the server
the moment a player is sold. No admin action needed.

WHAT HAPPENS AUTOMATICALLY ON SALE
------------------------------------
1.  Player.status           = sold
2.  Player.soldTo           = winning team ID
3.  Player.soldPrice        = winning bid amount
4.  Player.soldAt           = current timestamp
5.  Team.remainingBudget    -= soldPrice (atomic)
6.  Team.players            push playerId
7.  Team.squadSize          += 1
8.  Team.teamValue          += soldPrice
9.  Team.avgRating          recalculated
10. BidHistory entry        created for winning bid
11. Emit player:sold        to all in auction room
12. Emit budget:updated     to team:TEAM_ID private room

SOLD PLAYER FEED
-----------------
Scrollable feed below auction stage showing all sold players, most recent first.
Each entry shows: photo, name, rating, position, buying team, sale price.

BUDGET DISPLAY RULES
---------------------
- Top of owner screen: persistent budget bar, Spent / Remaining / Total
- Updates real-time via budget:updated socket event
- Warning when remaining less than 20% of total (amber color)
- Critical when remaining less than 100 (red flash animation)

PART 24 - COMMON PITFALLS AND FIXES
================================================================================

Problem                         Cause                       Fix
-----------------------------   -------------------------   ------------------------------------------
Timer drifts between clients    Timer runs on client        Timer MUST run on server only
Two bids accepted at once       No race condition fix        Use findOneAndUpdate with $lt conditional
Budget goes negative            No server validation        Check remainingBudget >= amount before bid
Player added to pool twice      No uniqueness check         Reject if status is not available
Socket auth bypass              No token validation         Validate JWT in io.use() middleware
Timer not stopping              setInterval leak            clearInterval on pause sell skip complete
Player images show 404          Dead URLs in database       Always implement fallback SVG avatar
Double bid on mobile            Tap registered twice        Disable bid button for 1 second after click
CORS error on frontend          Wrong FRONTEND_URL          Exact match between Railway env and Vercel URL
Railway deploy fails            TypeScript compile error    Run npm run build locally before pushing
Vercel build fails              Next.js type error          Run npm run build in /frontend locally first
Socket not connecting           Wrong SOCKET_URL            NEXT_PUBLIC_SOCKET_URL must match Railway exactly
Budget not deducting            Atomic update failing       Check MongoDB connection in Railway logs
